لدينا Bug خطير في دورة طلبات ترقية الاشتراك ويجب إصلاحه بشكل صارم وكامل من الجذر، وليس ترقيعًا بصريًا.

========================================
المشكلة الحالية
========================================

قام مستخدم بإرسال طلب ترقية اشتراك، وظهر له في الواجهة:
"تم إرسال طلبك بنجاح"

لكن عند الدخول إلى لوحة تحكم الأدمن > طلبات الاشتراك:
- لا يظهر الطلب الجديد بشكل صحيح كما يجب
- لا يظهر إشعار للأدمن بوجود طلب اشتراك جديد
- حالة الطلب المتوقعة يجب أن تكون "قيد المراجعة" أو "بانتظار المراجعة"
- هناك أيضًا احتمال وجود عدم تطابق في نوع الحساب أو اسم الباقة أو فلترة الحالة

مثال الحالة:
المستخدم: البراء
نوع الحساب المتوقع: مكتب عقاري
تم إرسال طلب ترقية
لكن لوحة الأدمن لا تعكس ذلك بشكل صحيح

========================================
الهدف
========================================

إصلاح دورة طلبات الاشتراك End-to-End بشكل صارم بحيث:

1) عند إرسال أي طلب ترقية/اشتراك من أي مستخدم:
- يتم إنشاء سجل SubscriptionRequest/PlanChangeRequest بشكل صحيح في قاعدة البيانات
- يتم ربطه بالمستخدم الصحيح
- يتم حفظ نوع الحساب الصحيح
- يتم حفظ الباقة المطلوبة بالكود الصحيح
- يتم حفظ الحالة الابتدائية بشكل موحد
- يتم رفع إشعار للأدمن فورًا

2) عند فتح لوحة الأدمن:
- يظهر الطلب الجديد مباشرة ضمن "طلبات الاشتراك"
- يظهر العداد/الإحصائيات بشكل صحيح
- يظهر الطلب ضمن حالة "قيد المراجعة" أو الحالة القياسية المعتمدة
- يظهر إشعار جديد للأدمن في نظام الإشعارات

3) يجب منع أي اختلاف بين:
- نوع المستخدم الظاهر في الواجهة
- نوع المستخدم المخزن في قاعدة البيانات
- نوع المستخدم المعتمد عند تصفية الطلبات في لوحة الأدمن

========================================
المطلوب تنفيذه
========================================

1. تتبع تدفق إنشاء الطلب من الواجهة حتى قاعدة البيانات
افحص المسار الكامل التالي:
- صفحة الباقات
- زر اختيار الباقة / ترقية الاشتراك
- المودال أو الفورم
- الـ API أو server action أو controller
- service/repository الخاص بطلبات الاشتراك
- قاعدة البيانات
- لوحة الأدمن
- نظام الإشعارات

أريد معرفة أين ينكسر التدفق بالضبط:
- هل الطلب لا يُحفظ؟
- هل يُحفظ لكن بحالة مختلفة؟
- هل يُحفظ لكن بنوع مستخدم خاطئ؟
- هل يُحفظ لكن لوحة الأدمن تستخدم فلترًا خاطئًا؟
- هل الإشعار لا يُنشأ أصلًا؟
- هل الإشعار يُنشأ لكن لا يُعرض للأدمن؟

2. توحيد الحالة الابتدائية للطلب
اعتمد enum/state موحدة وواضحة لطلبات الاشتراك، مثل:
- pending_review
- approved
- rejected
- paid
- payment_uploaded
- cancelled

ممنوع استخدام حالات مبعثرة أو مترادفات غير موحدة مثل:
- pending
- under_review
- waiting
- review
- new
إلا إذا كانت كلها مضمونة عبر mapping موحد

المطلوب:
- تعريف مصدر واحد للحالات
- استخدام نفس القيمة في DB + API + admin UI + filters + counters
- جعل الحالة الافتراضية عند إنشاء طلب جديد:
  pending_review

3. إصلاح فلترة لوحة الأدمن
افحص صفحة:
Admin > Subscription Requests

وتأكد أن:
- الطلبات الجديدة لا يتم استبعادها بسبب فلتر افتراضي
- العدادات في الأعلى تعتمد على نفس الحالات الحقيقية في DB
- قائمة الطلبات تعتمد على نفس query logic
- لا يوجد mismatch بين counters والجدول

المطلوب:
- توحيد query المصدر للعدادات والجدول
- إذا كانت الحالة pending_review يجب أن تظهر في:
  "قيد الانتظار" أو "قيد المراجعة" حسب التسمية المعتمدة
- إزالة أي mapping عشوائي بين النصوص العربية وقيم DB

4. إصلاح ربط نوع المستخدم
يوجد اشتباه أن الطلب أرسل من مستخدم لكن تم تصنيفه أو عرضه بشكل لا يتوافق مع نوع حسابه الحقيقي.

المطلوب:
- التحقق من userId
- التحقق من accountType / role / profileType
- التحقق من requestedPlanCode
- التحقق من currentPlanCode
- التحقق من target audience للباقة

ويجب منع الحالات التالية:
- مستخدم مكتب عقاري يرسل طلبًا فيُسجل كوسيط
- ظهور باقة BrokerPremium لحساب Office إذا كان هذا غير صحيح
- الاعتماد على label ظاهر فقط بدل internal plan code

5. توحيد model الباقات
افحص هل النظام ما زال يستخدم أسماء قديمة أو غير موحدة مثل:
- BrokerPremium
- Silver
- broker_premium
- office premium
- Business
- labels مترجمة غير ثابتة

المطلوب:
- الاعتماد فقط على internal plan code موحد
مثال:
- office_free
- office_basic
- office_pro
- broker_free
- broker_basic
- broker_pro
- company_enterprise

ويتم تخزين:
- requestedPlanCode
- currentPlanCode
- accountType
بشكل صريح في الطلب

6. إنشاء إشعار أدمن حقيقي عند الطلب
عند إنشاء أي Subscription Request جديد:
- أنشئ Notification للأدمن مباشرة
- نوع الإشعار: subscription_request_created
- اربطه بـ requestId
- اربطه بـ userId
- اجعل عنوانه واضحًا:
  "طلب اشتراك جديد"
- ووصفه:
  "قام المستخدم [الاسم] بإرسال طلب ترقية إلى الباقة [اسم الباقة]"

ويجب أن يظهر هذا الإشعار في:
- جرس الإشعارات للأدمن
- قائمة الإشعارات
- badge counter

7. دعم تعدد الأدمن
إذا كان هناك أكثر من أدمن:
- أرسل الإشعار لجميع المستخدمين المخولين بإدارة الاشتراكات
- أو على الأقل لجميع من لديهم permission مناسب مثل:
  manage_subscriptions
  review_payments
  admin_access

8. إصلاح صفحة "تم إرسال طلبك بنجاح"
حاليًا الواجهة توحي أن العملية نجحت حتى لو لم تكتمل الدورة بشكل صحيح.

المطلوب:
- لا تعرض رسالة النجاح إلا بعد التأكد 100% من:
  1) إنشاء الطلب في DB
  2) تعيين حالته الافتراضية الصحيحة
  3) إنشاء إشعار الأدمن
- إذا فشل أي جزء:
  - rollback أو failure handling واضح
  - أظهر رسالة خطأ صحيحة للمستخدم
  - لا تقل "تم إرسال طلبك بنجاح" إلا إذا اكتملت العملية بالكامل

9. Logging صارم
أضف logs واضحة في:
- إنشاء الطلب
- تعيين الحالة
- ربط المستخدم
- إنشاء الإشعار
- جلب الطلبات في الأدمن
- فلترة الحالة
- احتساب العدادات

أريد logs تساعد على معرفة:
- requestId
- userId
- accountType
- currentPlanCode
- requestedPlanCode
- status
- notificationCreated = true/false

10. Backfill / Repair للأرقام الحالية
إذا كانت هناك طلبات موجودة بالفعل في قاعدة البيانات لكنها غير ظاهرة بسبب خلل سابق:
- افحص الطلبات الحالية
- أصلح الحالات غير الموحدة عبر migration أو repair script
- أصلح plan codes القديمة إن وجدت
- أصلح الإشعارات الناقصة إن أمكن

========================================
اختبارات إلزامية
========================================

نفّذ واختبر السيناريوهات التالية:

1) مستخدم Broker يرسل طلب ترقية
- يُنشأ الطلب
- حالته pending_review
- يظهر في الأدمن
- يظهر إشعار

2) مستخدم Office يرسل طلب ترقية
- يُنشأ الطلب
- يرتبط بنوع الحساب الصحيح
- يظهر في الأدمن
- يظهر إشعار

3) مستخدم Company يرسل طلبًا
- لا يختلط مع أنواع أخرى
- يظهر بشكل صحيح

4) رفع إثبات دفع
- يغير الحالة بشكل صحيح
- يظهر في الأدمن
- ينشئ إشعارًا مناسبًا

5) عدادات صفحة الأدمن
- total
- pending_review
- approved
- rejected
كلها تطابق السجلات الحقيقية

6) قائمة الطلبات
- تعرض نفس نتائج العدادات
- لا يوجد mismatch

========================================
مخرجات مطلوبة مني
========================================

بعد التنفيذ أريد منك تقريرًا واضحًا يحتوي على:

1) السبب الجذري الحقيقي للمشكلة
2) الملفات التي عدلتها
3) ما الذي كان يمنع ظهور الطلب في لوحة الأدمن
4) ما الذي كان يمنع ظهور الإشعار
5) كيف وحدت الحالات
6) كيف وحدت plan codes وأنواع المستخدمين
7) نتيجة الاختبارات
8) هل تم إصلاح الطلبات القديمة أيضًا أم فقط الطلبات الجديدة

========================================
شرط مهم جدًا
========================================

ممنوع الحلول الشكلية فقط.

لا أريد:
- تعديل نصوص فقط
- إخفاء المشكلة من الواجهة
- إضافة toast نجاح بدون تحقق
- تعديل فلتر واحد وترك بقية النظام غير متناسق

أريد إصلاحًا جذريًا كاملًا على مستوى:
DB + backend + admin queries + notifications + state model + plan mapping

ابدأ الآن بالتشخيص أولًا، ثم طبّق الإصلاح الكامل، ثم أعطني تقريرًا نهائيًا دقيقًا.
