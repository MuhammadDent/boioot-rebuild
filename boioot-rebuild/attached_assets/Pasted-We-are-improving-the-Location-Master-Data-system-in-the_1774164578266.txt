GOAL

==================================================
NORMALIZATION STRATEGY
==================================================

Create reusable normalization utilities.

We need TWO normalization functions:

1) Strict normalization
Used for:
- real duplicate prevention
- normalized value stored in database
- exact duplicate checks

2) Soft normalization
Used only for:
- smart suggestions
- detecting likely similar values before create

==================================================
STRICT NORMALIZATION RULES
==================================================

Implement a reusable function like:
normalizeArabicLocationName()

Rules:
- trim leading and trailing spaces
- collapse multiple spaces into one
- remove Arabic diacritics
- remove tatweel/kashida
- normalize Arabic alef variants:
  أ / إ / آ -> ا
- normalize ى -> ي
- convert Arabic digits to English digits
- replace separators/punctuation like:
  - _ / \ . , ( ) [ ] { } |
  with spaces
- lowercase English letters
- keep ة unchanged
- keep ال unchanged
- after replacements, collapse spaces again and trim

IMPORTANT:
Strict normalization must be the source of NormalizedName.

==================================================
SOFT NORMALIZATION RULES
==================================================

Implement a second reusable function like:
softNormalizeArabicLocationName()

It should start from the strict normalized result,
then apply extra relaxed comparison rules only for suggestions.

For example:
- optionally remove leading "ال" for suggestion matching only
- do NOT use soft normalization for uniqueness constraints
- do NOT save soft normalized value as the main DB normalized value

Soft normalization is for:
"Did you mean this existing location?"

==================================================
DISPLAY NAME VS NORMALIZED NAME
==================================================

Always store both:

1) DisplayName
- the user-facing visible value
- shown in dropdowns and UI

2) NormalizedName
- generated automatically from DisplayName using strict normalization
- used for duplicate checks and DB uniqueness

Do NOT rely on DisplayName alone for duplicate prevention.

==================================================
CREATE FLOW RULES
==================================================

When a user adds a new city or district:

1) Take the raw input
2) Create DisplayName from trimmed user input
3) Create NormalizedName using strict normalization
4) Validate the parent exists:
   - city requires GovernorateId
   - district requires CityId
5) Check exact duplicate using:
   - City: same GovernorateId + same NormalizedName
   - District: same CityId + same NormalizedName

If exact duplicate exists:
- do NOT create a new record
- return the existing record
- return a friendly message like:
  "This location already exists"

6) If no exact duplicate exists:
- run soft normalization comparison against sibling records
- if similar entry is found:
  return suggestion like:
  "Did you mean: الهفوف ؟"

7) Only create a new record if:
- no strict duplicate exists
- and no suggestion is selected instead
