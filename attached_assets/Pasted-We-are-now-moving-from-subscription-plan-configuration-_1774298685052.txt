We are now moving from subscription plan configuration
to real runtime enforcement.

Current state:
- Plans exist
- FeatureDefinitions exist
- LimitDefinitions exist
- PlanFeatures exist
- PlanLimits exist
- Admin can configure plans

GOAL:
Turn the plan system into a real enforcement engine that blocks disallowed actions at runtime.

This must be backend-driven and production-safe.
Frontend can reflect restrictions, but backend must be the final authority.

==================================================
PHASE 1 — AUDIT ENFORCEMENT ENTRY POINTS
==================================================

First inspect the current application flows and identify all places where subscription limits and features should be enforced.

At minimum inspect these flows:

1. Listing / Ad creation
2. Listing / Ad update
3. Image upload for listing
4. Video upload for listing
5. Request creation / request receiving
6. Internal messaging / chat
7. Analytics dashboard access
8. Featured / promoted listing options
9. Any premium-only tools in user dashboard

Return a short audit:
- endpoint / service / handler
- what should be enforced there
- current state (not enforced / partially enforced / already enforced)

Do not guess.
Trace actual backend paths.

==================================================
PHASE 2 — BUILD A SINGLE SOURCE OF TRUTH
==================================================

Create a backend entitlement service and use it everywhere.

Suggested interface:
- ISubscriptionEntitlementService

Suggested methods:
- GetUserSubscriptionAsync(userId)
- HasFeatureAsync(userId, featureKey)
- GetLimitAsync(userId, limitKey)
- GetEntitlementsAsync(userId)
- EnsureFeatureEnabledAsync(userId, featureKey)
- EnsureWithinLimitAsync(userId, limitKey, currentValue, requestedValue = 1)

Important:
- Read from DB only
- No hardcoded package logic
- No role-based bypasses except explicit admin superuser rules if already approved
- Keep logic centralized, reusable, and testable

==================================================
PHASE 3 — ENFORCEMENT RULES
==================================================

Implement runtime enforcement rules for the following:

A) Listing creation
Examples:
- max_active_listings
- max_total_listings if applicable

Behavior:
- Before creating a listing, count the user's currently active listings
- Compare against plan limit
- If limit exceeded, block creation with a clear business error

B) Image upload per listing
Examples:
- max_images_per_listing

Behavior:
- Count existing images on the listing
- If uploading more would exceed limit, block request

C) Video upload
Use both feature + limit:
- feature: video_upload
- limit: max_videos_per_listing

Behavior:
- If video_upload = false -> reject any video upload
- If enabled, enforce max_videos_per_listing
- If limit is 0, reject upload
- Validate at backend, not just frontend

D) Requests
Examples:
- max_requests
- max_received_requests if your architecture distinguishes them

Behavior:
- Enforce according to your current request model
- If not fully clear, document your interpretation before coding

E) Internal messaging
Examples:
- internal_chat
- max_messages or max_conversations if present

Behavior:
- If chat feature disabled, block message sending / conversation creation
- If numeric limits exist, enforce them

F) Analytics access
Feature:
- analytics_dashboard

Behavior:
- If disabled, block analytics endpoint access
- Return appropriate forbidden/business response

G) Featured / promoted listing options
Features such as:
- featured_listing
- priority_listing
- homepage_boost
or equivalent if already in catalog

Behavior:
- If feature not enabled, reject premium promotion usage

PHASE 4 — BUSINESS ERROR MODEL
==================================================

Create clean business/domain errors for enforcement failures.

Examples:
- PlanFeatureDisabledException
- PlanLimitExceededException

API responses should be user-friendly and structured.

Examples:
- code: FEATURE_DISABLED
- message: Your current plan does not allow video uploads.
- details:
  - featureKey: video_upload

and

- code: LIMIT_EXCEEDED
- message: You have reached your plan limit for active listings.
- details:
  - limitKey: max_active_listings
  - allowed: 20
  - current: 20
  - requested: 1

Do not return vague 500 errors.
